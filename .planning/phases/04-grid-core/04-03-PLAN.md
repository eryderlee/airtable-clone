  <files>

  <action>
    **Review and apply these performance optimizations if not already present:**

    **1. Scroll container `contain: strict`:**
    Ensure the scroll container div has `style={{ contain: "strict" }}` — this tells the browser to isolate layout/paint calculations to this container, preventing layout thrash on scroll.

    **2. Row height consistency:**
    Ensure `estimateSize: () => 32` matches actual rendered row height. If rows are taller (e.g., due to padding), update the estimate. Mismatched estimates cause scroll jumps.

    **3. Overscan tuning:**
    - Current: `overscan: 20` — this renders 20 extra rows above and below the viewport
    - If scroll feels smooth, keep at 20. If there's visible blank space during fast scroll, increase to 30
    - If performance is sluggish, decrease to 10

    **4. Verify memoization:**
    - `columnDefs` MUST be in `useMemo` (already done in 04-01)
    - `flatRows` MUST be in `useMemo` (already done in 04-01)
    - Verify neither is recreated on every render by checking dependency arrays

    **5. useInfiniteQuery staleTime:**
    Add `staleTime: 60_000` (1 minute) to the infinite query options to prevent unnecessary refetches when switching tabs/views and coming back:
    ```typescript
    api.row.getRows.useInfiniteQuery(
      { tableId, viewId, limit: 100 },
      {
        getNextPageParam: (lastPage) => lastPage.nextCursor ?? undefined,
        refetchOnWindowFocus: false,
        placeholderData: keepPreviousData,
        staleTime: 60_000,
      }
    );
    ```

    **6. Prevent cell re-render on scroll:**
    Verify that `flexRender(cell.column.columnDef.cell, cell.getContext())` is used (the default cell renderer). Do NOT wrap cells in custom components that create new closures on every render.

    **7. Row count display accuracy:**
    The toolbar currently shows `flatRows.length` which only counts loaded rows. This is correct behavior — we do NOT run COUNT(*) on 1M rows. The count increments as more pages load. If the user has loaded 500 of 1M rows, it shows "500 rows". This is intentional.

    **8. Verify scroll threshold triggers correctly:**
    The 500px threshold in `handleScroll` should trigger `fetchNextPage` before the user reaches the end of loaded data. With 100 rows per page at 32px each = 3200px of content, and 500px threshold, fetchNextPage fires at ~2700px scroll position — well before the end.

    **9. Add React.memo to GridTable if not already:**
    Wrap the GridTable component export in `React.memo` to prevent re-renders when parent (GridView) re-renders due to unrelated state changes:
    ```typescript
    export const GridTable = React.memo(function GridTable({ ... }: GridTableProps) {
      // component body
    });
    ```

    **10. Verify `getRowId` is set:**
    `getRowId: (row) => row.id` on useReactTable — this prevents TanStack Table from using array indices, which break when data changes (page loads, mutations).
