<task type="checkpoint:human-action" gate="blocking">
  <name>Task 1: Export data from Supabase and restore to Neon</name>
  <action>
    The user must run pg_dump/pg_restore commands because they require Supabase credentials that Claude does not have access to. The Neon DIRECT_URL from Plan 09-01 is needed for the restore target.
  </action>
  <instructions>
    **Option A: pg_dump/pg_restore (preferred — preserves all data)**

    Step 1: Verify pg_dump is installed locally
    ```bash
    pg_dump --version
    ```
    If not installed, download PostgreSQL client tools from https://www.postgresql.org/download/windows/ (match major version to Supabase: likely 15 or 16). Only the client tools are needed, not the full server.

    Step 2: Get Supabase session-mode pooler URL
    - Go to Supabase Dashboard -> Project Settings -> Database -> Connection Pooling
    - Switch mode to "Session" (NOT Transaction)
    - Copy the connection string — it should use port 5432 on `*.pooler.supabase.com`
    - Format: `postgresql://postgres.[ref]:[password]@aws-0-[region].pooler.supabase.com:5432/postgres`

    Step 3: Test connection first (fast, schema-only)
    ```bash
    pg_dump --schema-only -d "YOUR_SUPABASE_SESSION_URL" --schema=public > /dev/null
    ```
    If this hangs for more than 30 seconds, skip to **Option B** below.

    Step 4: Full dump
    ```bash
    pg_dump -Fc -v \
      -d "YOUR_SUPABASE_SESSION_URL" \
      --schema=public \
      --no-acl \
      -f supabase_dump.bak
    ```
    This may take a few minutes for 1M rows.

    Step 5: Restore to Neon (use your DIRECT_URL from .env — the one WITHOUT -pooler)
    ```bash
    pg_restore \
      -d "YOUR_NEON_DIRECT_URL" \
      -v --no-owner --no-acl \
      supabase_dump.bak
    ```
    Note: You may see warnings about "table already exists" if drizzle-kit push already created the schema — these warnings are safe to ignore. If you get actual errors about duplicate data, you may need to drop and recreate the Neon tables first by running `npx drizzle-kit push --force` or dropping tables via the Neon SQL editor.

    **Option B: Re-seed from scratch (fallback — if pg_dump fails or hangs)**

    If pg_dump fails (port blocked, timeout, SSL error), skip data migration entirely:
    1. The schema is already applied from Plan 09-01 (drizzle-kit push)
    2. Start the dev server: `npm run dev`
    3. Sign in and create a base/table through the UI — this confirms the app works
    4. The 1M-row seed can be re-created later via the "Add 100k rows" button (10 clicks) or by running the seed script

    Tell Claude which option you completed (A or B) and whether you saw any errors.
  </instructions>
  <resume-signal>Tell Claude: "Option A complete" or "Option B — re-seeded" (and any errors encountered)</resume-signal>
</task>
